## Rules

### 1. Do not parallel-batch a query whose siblings can exit non-zero

95 cancelled parallel calls across 29 sessions (down from 115 / 32 last
run — the `parallel-safe-queries.md` rule is starting to land). The
mechanism is unchanged: when one Bash/Read/Glob call in a parallel batch
exits non-zero, every sibling is marked cancelled.

Top offenders this week (the *first* call to die that took its siblings
with it):

| Stem | Cancelled-sibling events |
|---|---|
| `task project:skullcaps-native status:pending list` | 19 |
| `tar -xzf $HOME/repos/ForumVirium/…` | 18 |
| `ls $HOME/Downloads/Skullcaps/…` | 5 |
| `notebooklm source delete <id>` | 4 |
| `./build/skullcaps terrain info …` | 3 |
| `task project:skullcaps-native next limit…` | 3 |
| `Read($HOME/Downloads/Skullcaps/…)` | 3 |
| `cd $HOME/repos/laurigates/claude…` | 3 |
| Various `jq` pipelines on possibly-empty input | 8+ |
| `kubectl --context=… get …` | 2 |
| `SHA=$(git rev-parse --short HEAD); task …` | 2 |
| `grep -rn 'Point(' --include="*.py" …` | 2 |

The remediation is `parallel-safe-queries.md` (already in
`claude-plugins/.claude/rules/`):

```
# Wrong — `task list` exits 1 on empty
[parallel batch]
  Bash("task project:foo status:pending list")
  Bash("task project:foo status:active list")
  Bash("task project:foo status:done list")

# Correct — `export | jq` is always exit-0
[parallel batch]
  Bash("task project:foo status:pending export | jq '.[]'")
  Bash("task project:foo status:active export | jq '.[]'")
  Bash("task project:foo status:done export | jq '.[]'")
```

For `tar`, `ls <glob>`, and Read-on-missing-file, the fix is **verify
existence first** (single-call `Glob` or `ls -1`), then issue the
parallel batch only over confirmed-present paths.

### 2. Verify the path exists before `Read`

53 events across 38 sessions (unchanged from last run). **Every** event
this week carried the "current working directory is X" hint, confirming
that the dominant cause is assumed cwd: the session's working directory
had moved (worktree switch, `cd` in a prior turn, agent thread reset)
and the cached path no longer resolved.

**Do**:

```
# Confirm before reading
Glob(pattern="/abs/path/to/file.md")     # returns [] if missing
# Or, when scanning a directory:
Bash("ls -1 /abs/path/to/dir 2>/dev/null || echo MISSING")
```

**Do not** chain three Read calls against guessed paths; one Glob covers
all of them in a single call and tells you which exist.

Agent-thread cwd note (from this agent's own system prompt): "Agent
threads always have their cwd reset between bash calls" — always pass
absolute paths, never assume the cwd is preserved.

### 3. `Read` is for files; use `Glob` or `ls` for directories

42 EISDIR errors across 17 sessions (unchanged). The Read tool refuses
directories with `EISDIR: illegal operation on a directory`. The error
message does not suggest the alternative, so the agent often retries
with a slightly different path before giving up.

```
# Wrong
Read(file_path="/abs/path/to/skills")
# → EISDIR

# Correct — list contents
Glob(pattern="/abs/path/to/skills/**/*.md")
# Or
Bash("ls -1 /abs/path/to/skills")
```

### 4. Always `Read` before `Edit` or `Write` (per session)

31 events across 13 sessions (down slightly from 35 / 16 last run). The
harness tracks file-read state **per session**. Reading the file in a
previous Claude Code session does not satisfy the requirement. The
error message is clear:

> File has not been read yet. Read it first before writing to it.

Pattern: at the start of an editing turn, batch the Reads for every file
you intend to touch. Then do the Edits. Do not interleave a Read for
file A immediately before its Edit while having batched the Edits for
files B, C, D — you will hit "not been read yet" on B, C, D.

### 5. NEW — `Read` refuses files larger than 25 000 tokens

32 events across 29 sessions. The Read tool's error is explicit:

> File content (164 836 tokens) exceeds maximum allowed tokens
> (25 000). Use offset and limit parameters to read specific portions
> of the file, or search for specific content instead of reading the
> whole file.

This week's distribution: average refused size 52 000 tokens, max
200 000. Common offenders: vendored JSON dumps, generated schema
files, lockfiles, transcript files, and large generated docs.

**Do**:

```
# Look up a specific location with Grep first
Grep(pattern="needle", path="/abs/path", output_mode="content", -n=true)

# Then Read just that range with offset+limit
Read(file_path="/abs/path", offset=420, limit=80)
```

**Do not** Read a multi-megabyte file "to inspect it" — paginate
through with `offset`/`limit` (max ~2000 lines per call) or search for
the specific content you need.

### 6. NEW — Re-Read before Edit when a formatter or hook may have run

7 events across 7 sessions. The error:

> File has been modified since read, either by the user or by a linter.
> Read it again before attempting to write it.

This is **distinct** from rule #4 (Edit-without-Read). Here the file
*was* read this session, but a formatter (`prettier`, `stylua`,
`ruff format`), pre-commit hook, build watcher, or concurrent coworker
rewrote it between your `Read` and your `Edit`.

Triggers to watch:

| After this happens… | Re-Read before next Edit |
|---|---|
| You ran `pre-commit run` | All staged files |
| You ran a formatter (`prettier --write`, `stylua`, `ruff format`) | Files in scope |
| You committed (commit hooks may have rewritten files) | Files just committed |
| A coworker agent is detected (see `agent-coworker-detection.md`) | All in-flight files |
| A long Bash command ran in the background while you were editing | The files it wrote |

The fix is mechanical: when the harness reports "modified since read,"
do not retry the Edit — issue a fresh Read first, then re-craft the
Edit against the new line numbers.

### 7. Configure pre-commit hooks for empty input

53 events across 53 sessions (sessions correctly counted this run; was
"51 / 1+" before). 49 of those 53 are the chezmoi shellcheck hook
emitting:

> shellcheck...........................................(no files to check) Skipped

Two fixes for `.pre-commit-config.yaml` entries:

```yaml
# Option A — tighten the files: regex so the hook isn't selected at all
- id: shellcheck
  files: \.(sh|bash)$
  pass_filenames: true

# Option B — wrap with a script that no-ops on empty input
- id: shellcheck
  entry: bash -c 'test -n "$1" && shellcheck "$@" || true' --
  pass_filenames: true
```

This is a config rule, not an agent-behavior rule — flag it the next
time you touch the pre-commit config in chezmoi, dotfiles, or any repo
that surfaces it. The chezmoi config is the dominant offender (49 of
53 events).

### 8. Check for an existing PR before pushing to a feature branch

3 events across 2 sessions (down from 8 / 2). Hook is informational,
not blocking — but signals that the agent is about to redo work it
can skip:

> PR METADATA CHECK: You are pushing to a branch with an existing PR.
> PR #N: <title>

```
gh pr list --head "$(git branch --show-current)" --json number,title,url --jq '.[]'
```

If a PR exists, push (the hook permits it) and **do not** call
`gh pr create` afterward.

### 9. WebFetch — do not retry the same failing URL

12 events across 5 sessions (unchanged). This week split between 404
(stale `?ref=…` paths) and 403 (auth-required / rate-limited). When
a fetch fails:

| Failure | Try |
|---|---|
| 404 on a docs page with `?ref=foo` | Strip query string |
| 404 on a GitHub blob URL | Switch to `raw.githubusercontent.com` URL |
| 404 on a repo path | `gh api repos/<owner>/<repo>/contents/<path>` |
| 403 on a public docs page | Try once with a different user agent or a search engine |
| 403 on a GitHub URL | Use `gh api` (authenticated) |
| Timeout | One retry, then fall back to context7 / WebSearch |

If two attempts both fail, surface the failure in the response — do
not loop.

### 10. `helm` and `kubectl` need an explicit context flag

3 hook blocks across 2 sessions (unchanged). The flag-hook is blocking,
not advisory.

```
# Wrong
helm upgrade myrelease ./chart
kubectl get pods

# Correct
helm upgrade myrelease ./chart --kube-context fvh-staging
kubectl --context fvh-staging get pods
```

Discover the right context with `kubectl config get-contexts -o name`
first.

### 11. Use `task export | jq` — not `task list` — in any parallel context

Sub-rule of #1. `task <filter> list` exits 1 when the filter matches
zero tasks. In a parallel batch this kills every sibling tool call.

```
# Cancels siblings on an empty result
Bash("task project:skullcaps-native status:pending list")

# Always exit-0
Bash("task project:skullcaps-native status:pending export | jq -r '.[].description'")
```

This rule is documented at length in
`claude-plugins/.claude/rules/parallel-safe-queries.md`.

### 12. Treat "Permission to use Bash has been denied" as terminal

36 events across 4 sessions (unchanged), still concentrated in two
sessions where the agent retried the same denied command repeatedly
(17 events in one session, 9 in another). The harness permission-denied
message is a final answer for *this command*. The correct moves:

1. Read the suggestion in the denial message ("you may attempt to
   accomplish this action using other tools…")
2. If a viable alternative exists, use it.
3. If not, explain to the user why the command is needed and provide
   the exact command to run manually (template in
   `handling-blocked-hooks.md`).

Do not retry the same command with cosmetic variations (different
quoting, prepended `env`, etc.) — it will be denied again.

### 13. NEW — Stop entering plan mode for Q&A and routine work

15 `ExitPlanMode` calls were rejected by the user across 14 sessions —
nearly 3× the entered-plan-mode count (5 / 4). The rule already exists
(`plan-mode.md`) and the W16 report flagged this at volume 30 / many.
The numbers are down (good) but the rejection ratio is climbing —
when plan mode is invoked, the user is rejecting it most of the time.

Re-read `plan-mode.md`. Only enter plan mode when:

- The user explicitly asks for a plan ("plan out X", "show me a plan
  before acting")
- The change is large enough that confirming the approach prevents
  costly rework (≥ 3 distinct files / multiple subsystems / hard-to-revert)

For Q&A, observational prompts ("analyze these findings"), and
execution requests ("run X"), respond directly.
